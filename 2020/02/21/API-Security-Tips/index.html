发表于 2020-02-21 | 分类于 Web安全

注意不同版本的API

通常较旧的API版本更容易遭受攻击。
可以利用不同版本接口的相似性、可预测性来查找旧版本进行测试。比如fuzz版本号。

比如：

遇到一个登录接口：api/v3/login？
可以很轻易的猜测到可能存在

较旧的版本可能没有太多安全机制。可能没有验证码和锁定机制，可以暴力破解，甚至可能存在SQL注入，任意用户登录等。

新的版本可能处于测试阶段，主要的登录功能已经完成，但是验证码等安全机制可能还没有加。

注意可以实现同一个功能的多个API

同一个系统，可能存在实现同一功能的多个api。

比如：

举一反三，子域名、子系统中也可能有实现主站功能的接口，而其安全性可能更差。

BOLA(IDOR)

不安全的对象引用（越权）

细心关注api中的各个对象参数（各种id等），熟悉各个业务的权限划分。

https://medium.com/@inonst/a-deep-dive-on-the-most-critical-api-vulnerability-bola-1342224ec3f2

RoR 命令注入tips

测试Ruby on Rails应用，并注意到一个包含URL的HTTP参数时，可以尝试命令注入。因为开发人员有时会使用“Kernel#open”函数来访问URL，只需发送管道作为第一个字符，然后发送命令即可实现命令注入。

About the open function: https://apidock.com/ruby/Kernel/open

不止局限于一端

不要以为 移动/Web 端使用的API使用了同样的安全机制。

测试完api/login之后，不妨测试一下api/mobile/login，可能会有新的发现。

检查API是否也支持SOAP

将 content-type 修改为 application/xml ,在请求正文中添加一个简单的XML，然后查看API如何处理它。

有时，身份验证是在REST和SOAP API之间共享的不同组件中完成的== SOAP API可能支持JWT。

先关注请求正文中的id

在测试越权漏洞的时候，HTTP请求正文中的ID往往比URL中的ID更容易受到攻击（当然，url中的id也别忘了测试）。

如果API使用了Authorization header，就不要考虑CSRF了。

即使id不可预测也要尝试

测试越权的过程中，即使id不可预测，或者是GUID而非数字，也要尝试发送数字值进行测试。因为有时会同时支持这两种方式。

扩大攻击面

查找子域

静态资源

对静态资源的权限校验可能不够严格，可以尝试越权查看其他用户的静态资源。

Burp

burp一直开着，target会以树状的形式记录访问过的所有API端点。

旧版本的移动客户端

遇到移动端证书锁定？尝试测试它们的旧版本，旧版本没有证书锁定的概率很大，节约时间。

旧版本中也可能会有一些旧版本的api。

关注边缘功能

测试人员和开发人员都会在主要功能点投入大量精力。我们可以尝试关注一些边缘的功能api，发现问题的概率很高。

容易出现问题的功能点

  • 用户组管理
  • 导出为CSV/HTML/PDF
  • 仪表板的自定义视图
  • 子用户的创建和管理
  • 对象共享（照片、帖子等）
  • …

非生产环境、非主站

主站的功能不一定只在主站上有，测试站点说不定还正常运行，子系统说不定也有同样的功能。测试站点和子系统的安全性可能更差。

http://www.anandpraka.sh/2016/03/how-i-could-have-hacked-your-facebook.html

关注JS中的api

通过寻找JS中的接口来扩大攻击面。
JS中有可能会有一些接口的访问方式等等，甚至会有更敏感的信息。

数据暴露过多

有些API可能泄漏用户不应该访问的过多数据。或者比如遇到api/receipt，可以尝试寻找类似

黑盒到白盒

如果发现了可以任意文件读取、下载的漏洞，可以获取源码之后，审计源代码寻找新的漏洞。

越权绕过

在某些情况下，AuthZ机制需要使用纯字符串（在这种情况下为ID），并且如果它接收到JSON，则不会执行AuthZ检查。

XSS

API不返回HTML，而是返回JSON。

.net 应用tips

遇到包含文件路径或名称的参数，开发人员有时会使用“ Path.Combine（path_1，path_2）”创建完整路径。Path.Combine具有一个奇怪的特性：如果param＃2是绝对路径，则param＃1被忽略。

了解应用的基础实现

有的api会公布应用程序的基础实现，或者我们从某些途径拿到实现文档之类的材料，就可以利用这些更好地了解用户，角色，资源及其之间的相关性，从而能发现更多更酷的漏洞。

旧版本的移动端

如果应用有移动客户端，尝试关注旧版本的客户端和旧的功能，旧版本可能安全性较弱。

limit dos

找到了limit/page 参数，比如/api/news?limit=100

它可能容易受到DoS。尝试发送长值（例如limit=999999999）

这种是因为超长字段进入了refer里，refer过长导致服务器无法响应。

类似的，超长参数传入了set-cookie里，也可以导致服务器无法响应。
